# ADR-008: `agent` tool calls bypass edge-based input gathering entirely

**Status:** Accepted
**Date:** 2026-07-17
**Related:** SPEC-008 §4, §5

## Context

Every node type built before this spec receives its inputs via graph edges, resolved by the engine before `execute()` ever runs — the engine can compute this statically because the data-flow shape is fixed at graph-authoring time. SPEC-008's `agent` node needed to let a model call other nodes as tools, but a tool call's target and arguments are decided *by the model, at runtime* — there is no fixed edge to resolve, because the choice doesn't exist until the model makes it. This is a genuinely different execution shape from anything the engine (or ADR-003's decoupling) had to accommodate before, and needed a deliberate decision about how far the exception should extend.

## Decision

- **A node referenced as a tool is invoked directly**: the `agent` node's own `execute()` calls the target node's `NodeDefinition.execute()` directly, passing the model-supplied arguments as `inputs`, bypassing the engine's edge-based gathering (`_run_graph_async`'s `gather_inputs`) entirely for that call. The target node still produces a normal `TraceRecord`, which the agent attaches as one of its own `child_traces` entries.
- **A tool-referenced node must not also receive normal incoming edges.** Enforced as a real validation rule (`check_agent_tool_references`, `backend/validation/rules.py`), not just documented convention: an unknown tool reference, or a referenced node with any incoming edge to any of its input slots, is a validation error. `check_required_inputs` is taught to skip a tool-referenced node's own required-input check (`agent_tool_referenced_ids`) — such a node's inputs are legitimately supplied at runtime by the agent's tool calls, not edges, so the normal "every required input needs an edge" rule would otherwise misfire on every valid tool node.
- **The concrete plumbing**: `run_graph()` (`backend/execution/engine.py`) injects one new, engine-owned key into `resources` before every run — `resources["nodes_by_id"] = {n.id: n for n in graph.nodes}` — so `agent.execute()` can look up a sibling node's full `NodeSpec` by id. This is injected unconditionally, for every run, not gated on "does this graph contain an agent node." It's set inside `run_graph()` itself (not by the CLI/API caller) so it's automatically correct and fresh at every level of nesting, including inside `loop`'s recursive `run_graph()` call — which reuses the outer `resources` dict, so without this living inside `run_graph()`, a nested loop iteration would see the *outer* graph's `nodes_by_id` instead of its own sub_graph's.
- **Scope: this is permanent, and deliberately narrow.** It exists solely to let `agent` invoke tool nodes directly; it is not a general precedent for any node type to reach into sibling nodes, and no other node type does. `nodes_by_id` exposes node lookup by id only — no edges — reinforcing that a tool call is not, and must never become, a way to route around the normal edge-based flow for anything else. `agent.execute()` itself further restricts what it will call to exactly the ids listed in its own `config.tools`; `nodes_by_id` being broader than that is an implementation convenience, not permission for the agent to call anything else in the graph.

## Rationale

- **Direct invocation, not a fake/synthetic edge, because there is nothing to synthesize.** An edge represents a fixed producer→consumer relationship known at graph-authoring time. A tool call's "producer" (the model's decision) doesn't exist until the agent is already running — there is no moment before execution when a real edge could be drawn. Modeling this as an edge would be a fiction the validator and scheduler would have to specially accommodate everywhere, for a relationship that isn't actually edge-shaped.
- **Validating the "no conflicting edges" constraint, rather than just documenting it**, because the failure mode if it silently went unenforced is genuinely confusing: a required-input node with no edge is simply marked `blocked` and skipped by the normal scheduler (existing, unrelated behavior since SPEC-001) — so a tool node that *also* had edges would either sit as a dead, never-firing normal node, or (if its inputs happened to be fully edge-satisfiable on their own) produce a second, completely independent execution and trace record outside the agent's control, alongside whatever the agent's own tool calls produced. Neither outcome is something a graph author would want; catching it at validation time, with a clear message naming the conflicting node, is much cheaper than debugging either failure mode after the fact.
- **Engine-injected, not caller-injected, specifically for correctness under recursion.** The alternative — having the CLI/API caller set `resources["nodes_by_id"]` once, before calling `run_graph` — looks equivalent for a top-level run, but breaks the moment a `loop` node (ADR from SPEC-004) recurses: `loop.execute_loop` calls `run_graph(iteration_graph, resources=ctx.resources)`, reusing the *outer* `resources` dict for the *inner* sub-graph run. If `nodes_by_id` were caller-set once, that inner run would incorrectly see the outer graph's node map instead of its own sub_graph's. Setting it inside `run_graph()` itself means every level of nesting gets a `nodes_by_id` that's correct for the graph actually running at that level, with zero changes needed to `loop.py` or any other recursive caller.
- **Unconditional injection (not gated on "does this graph have an agent node") keeps the engine free of node-type awareness**, consistent with ADR-003's decoupling. The alternative — the engine scanning the graph for `agent` nodes before deciding whether to compute `nodes_by_id` — would be the exact kind of type-name-aware branching ADR-003 exists to prevent, for a saving (skipping one cheap dict comprehension the engine already does the equivalent of internally) that isn't worth the coupling.
- **Why this is a real, disclosed exception rather than something to minimize or hide**: `run_graph()`'s own docstring has stated since SPEC-002 that "the engine has no knowledge of what any node type needs... each node's execute() resolves its own dependencies" — read strictly, `resources` was always meant to be a caller-populated, engine-passthrough-only bag. This ADR is the record that one exception now exists, why it was necessary, and why it's narrower than it might first appear: the engine already computes `nodes_by_id` for its own scheduling every run; this exposes that *same* pre-existing structural data to node bodies, rather than the engine acquiring new domain-specific knowledge (it still has zero awareness of connections, models, or tool-calling — only of the graph's own node list, which it already needed).

## Consequences

- `git diff backend/execution/engine.py` for this spec is small but real (one line inside `run_graph()`), unlike SPEC-006/007's empty diffs — expected and disclosed here, not a regression of the genericity bar those specs established.
- Any future node type wanting the same capability (direct sibling-node invocation) can reuse `resources["nodes_by_id"]` for free — but should not be assumed to need it; this remains an `agent`-specific pattern until a second real use case demonstrates the same need.
- A tool node's trace only ever appears nested under the invoking agent's `child_traces` — never in the graph's top-level `trace` list — since it's never invoked by the normal scheduler (a validated graph guarantees it has no edges, so the scheduler always marks it `blocked` and skips it silently, exactly like any other edge-free required-input node).
- `check_required_inputs` now has a third skip condition (`tool_referenced_ids`) alongside `unregistered_ids` — a param addition to an existing, internal-only rule function, not a public API change.

## Alternatives considered

- **Embedding a copy of each tool node's spec directly in the agent's own config** (mirroring how `loop.sub_graph` embeds a full nested `GraphSpec`): rejected — spec-008 §5 explicitly wants tool *references* (node ids in the same graph), not duplicated copies, specifically so a tool node's schema and its callability from an agent never drift apart. A copy would need to be kept in sync by hand.
- **Giving `ExecutionContext` the full `GraphSpec` (including edges) instead of just `nodes_by_id`**: rejected — exposing edges would blur the line this ADR is trying to keep sharp (tool calls are deliberately edge-free); node-lookup-by-id is the minimum surface `agent` actually needs.
- **Resolving tool nodes via the CLI/API layer, upstream, like named connections (ADR-007)**: rejected — connections are external, caller-scoped resources resolved once per run; tool nodes are graph-internal structure that must be looked up fresh, per tool call, by whatever id the model happens to choose at that moment. There's no fixed set to pre-resolve the way there is for named connections.
- **Not validating the "no conflicting edges" constraint at all, leaving it as a documented-only convention**: rejected — the failure modes if violated (a silently dead node, or a confusing duplicate independent execution) are exactly the kind of thing CLAUDE.md's "validate before execution" convention exists to catch early instead of at debug time.
